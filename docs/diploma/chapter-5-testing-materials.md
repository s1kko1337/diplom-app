фактически реализованного автоматизированного набора предлагается структура
из 5 разделов, лучше отражающая объём работы:

## 5.1 Выбор методологии и инструментов автоматизированного тестирования

### Уровни тестирования и обоснование выбранного

Промышленное программное обеспечение традиционно тестируется на четырёх
уровнях: модульное (unit), интеграционное, end-to-end (e2e) и приёмочное.
Для клиентской части системы мониторинга бурового оборудования, в которой
бо́льшая часть бизнес-логики сосредоточена в сценариях взаимодействия
пользователя с интерфейсом (создание нарядов на ТО, конструктор панелей
мониторинга, подтверждение критических уведомлений), наибольшую
доказательную ценность даёт e2e-тестирование. Оно проверяет сквозные
пользовательские пути в реальном браузере, включая работу маршрутизатора,
состояние Pinia-сторов, реактивность Vue-компонентов и интеграцию с
mock-runtime, эмулирующим серверную часть.

Дополнительно реализован минимальный «smoke»-уровень для нефункциональных
проверок (отрисовка в трёх ширинах экрана, поведение при потере связи,
размер touch-таргетов), а также кросс-браузерные прогоны.

### Обоснование выбора Playwright

В качестве фреймворка автоматизированного тестирования выбран Playwright
(версия 1.59.x), разработанный Microsoft. Альтернативы — Cypress и Selenium
WebDriver — рассматривались, но отвергнуты по следующим причинам:

— Playwright нативно поддерживает запуск тестов в трёх браузерных движках
(Chromium, Firefox, WebKit) из единой конфигурации, что покрывает
требование НФТ-9 о поддержке двух последних версий Chrome, Firefox и Edge.

— Встроенная утилита `webServer` поднимает Vite dev-сервер автоматически
перед прогоном тестов и переиспользует уже запущенный процесс в локальной
разработке — это устраняет необходимость в скриптах оркестрации.

— Возможность инжектировать произвольный JavaScript в каждую загрузку
страницы через `page.addInitScript` идеально подходит для подмены состояния
`localStorage` (фактического хранилища данных в проекте) ещё до того, как
SPA-приложение успеет прочитать его при инициализации.

— Богатая система фикстур с автоматической инъекцией зависимостей в
тестовые функции и встроенный механизм трассировок (`trace.zip`) для
post-mortem отладки падений в headless-режиме без необходимости поднимать
GUI.

— Поддержка JavaScript без обязательного TypeScript, что согласуется со
стеком приложения (Vue 3 + JavaScript).

### Стек инструментов

| Слой | Инструмент | Версия / назначение |
|------|-----------|---------------------|
| Тест-раннер | `@playwright/test` | 1.59.x — параллельный запуск, встроенные ассерции |
| Браузеры | Chromium / Firefox / WebKit | поставляются Playwright `playwright install` |
| Веб-сервер | Vite 7 (`npm run dev`) | стартует через `webServer` Playwright |
| Линтинг тестов | ESLint 9 + OxLint | расширены глобалы `node` + `browser` для `tests/e2e/**` |
| Изоляция состояния | `localStorage` mock-runtime | сидирование через `addInitScript` |
| Источник правды о состоянии | Pinia (DEV-экспозиция `window.__pinia`) | прямая инспекция сторов в тестах |

### Поддерживаемые сценарии запуска

В `package.json` добавлены четыре npm-скрипта:

— `npm run test:e2e` — полный прогон во всех трёх браузерах;
— `npm run test:e2e:ui` — интерактивный UI-режим Playwright для отладки;
— `npm run test:e2e:headed` — прогон с видимым браузером;
— `npm run test:e2e:report` — открытие сгенерированного HTML-отчёта.

Конфигурация (`playwright.config.js`) задаёт таймаут теста 30 секунд,
таймаут одной ассерции 7 секунд, две попытки в режиме CI, один в локальной
разработке, локаль `ru-RU` и часовой пояс `Europe/Moscow`. Включено
сохранение трассировок и видео для упавших тестов
(`trace: 'retain-on-failure'`, `video: 'retain-on-failure'`).

## 5.2 Архитектура тестового набора и приёмы изоляции состояния

### Структура каталогов

Тестовый набор организован по принципу разделения ответственности,
аналогичному production-коду:

```
tests/e2e/
├── fixtures/              # Расширения Playwright-test
│   ├── auth.js            #   loginAs(role) через addInitScript
│   ├── state.js           #   seedAlerts/Orders/Dashboard, resetMockState
│   └── index.js           #   объединённый test/expect с auto-фикстурами
├── helpers/               # Чистые утилиты без побочных эффектов
│   ├── selectors.js       #   ROLES, STATUS_LABELS, NAV_LINKS, TESTIDS
│   ├── storage.js         #   ключи rgm:*, makeAlert/makeOrder
│   └── viewport.js        #   пресеты 768/1440/1920/390
├── pages/                 # Page Object Model (14 классов)
│   ├── BasePage.js        #   общая шапка/сайдбар/connection-status
│   ├── LoginPage.js
│   ├── HomePage.js
│   ├── EquipmentListPage.js
│   ├── EquipmentDetailPage.js
│   ├── EquipmentDashboardPage.js
│   ├── AlertsPage.js
│   ├── MaintenanceListPage.js
│   ├── MaintenanceCreatePage.js
│   ├── MaintenanceDetailPage.js
│   ├── ReportsPage.js
│   ├── ReportCreatePage.js
│   ├── JournalPage.js
│   └── SettingsPage.js
└── specs/                 # 30 спек-файлов, 137 тестов (4 skip)
```

Общий объём тестового кода — около 2700 строк JavaScript (2081 строк
непосредственно в спеках, 405 — в POM-классах, 228 — в фикстурах и
хелперах).

### Page Object Model

Каждый ключевой экран (логин, список оборудования, детальная карточка,
конструктор панели, уведомления, наряды ТО, отчёты, журнал, настройки)
инкапсулирован в отдельный класс с локаторами и действиями. Это даёт три
преимущества:

1. **Селекторы централизованы.** При изменении в исходном коде названия
   кнопки или роли элемента правка делается в одном месте.

2. **Спецификации читаются как сценарии.** Тест выглядит как
   `await dash.configureButton.click(); await dash.addWidget('Числовой
   индикатор', 'Температура двигателя')` — без перегрузки Playwright-API.

3. **Повторное использование.** Один POM-класс задействован в нескольких
   спеках разной направленности (smoke, extended, ролевые сценарии).

Пример (фрагмент `EquipmentDashboardPage.js`):

```javascript
export class EquipmentDashboardPage {
  constructor(page, id) { this.page = page; this.id = id }
  async goto() { await this.page.goto(`/equipment/${this.id}/dashboard`) }
  get configureButton() { return this.page.getByRole('button', { name: 'Настроить' }) }
  get widgets() { return this.page.getByTestId('widget-card') }
  async addWidget(typeLabel, sensorLabel) {
    await this.addWidgetButton.click()
    await this.page.getByRole('button', { name: typeLabel }).click()
    if (sensorLabel) await this.page.getByRole('button', { name: sensorLabel }).click()
    await this.page.getByRole('button', { name: 'ДОБАВИТЬ' }).click()
  }
}
```

### Стратегия идентификации элементов

Приоритет селекторов: `getByRole` → `getByText` → `data-testid`. Доля
data-testid в коде минимизирована и вводится только там, где у элемента
нет естественной семантической роли:

— `connection-status` на индикаторе подключения в шапке;
— `widget-card` на обёртке виджета конструктора;
— `equipment-card` на карточке станка в списке;
— `alert-card` на карточке уведомления;
— `critical-alert-modal` на модальном окне критического оповещения.

Все остальные взаимодействия (нажатия кнопок, выбор пунктов меню,
ввод в поля) идут через ARIA-роли (`button`, `combobox`, `option`,
`tab`, `tabpanel`, `dialog`) с фильтрацией по видимому русскому тексту,
что одновременно служит косвенной проверкой доступности интерфейса.

### Фикстуры Playwright

Реализованы две специализированные фикстуры на основе `test.extend`:

— **`cleanState`** (auto-фикстура) — выполняется перед каждым тестом и
регистрирует `addInitScript`, который при первой загрузке страницы
очищает все ключи с префиксом `rgm:` и токены авторизации, а также
выставляет флаг `rgm:e2e:disable-thresholds=1`, выключающий пороговый
монитор (см. ниже про модальное окно). Чтобы повторные `page.reload()`
внутри теста не стирали данные, сохранённые тестом в процессе
исполнения, добавлен сторожевой флаг `__e2e_cleaned__` в `sessionStorage`.

— **`loginAs(role)`** — функциональная фикстура: вызов
`loginAs('engineer'|'mechanic'|'foreman')` инжектирует токен и
`auth_user_id` напрямую в `localStorage` через `addInitScript`. Это в
несколько раз быстрее логина через UI и устойчиво к редизайну формы
авторизации.

### Сидирование детерминированных данных

Для тестов, требующих заранее известного состояния (например, проверка
поведения при критическом уведомлении или приёмки наряда мастером),
определены три хелпера:

— `seedAlerts(page, alerts)` — инжектирует массив уведомлений в
`rgm:v1:alerts`;

— `seedOrders(page, orders)` — инжектирует наряды ТО в `rgm:v1:orders`;

— `seedDashboard(page, userId, config)` — инжектирует пользовательскую
раскладку панели мониторинга в `rgm:v1:user:<id>:dashboards`.

Для удобства каждый seed сопровождается фабрикой `makeAlert` /
`makeOrder` с осмысленными значениями по умолчанию и точечной
переопределяемостью через `overrides`.

### Доступ к Pinia из тестов

В DEV-сборке (`src/main.js`) добавлено условное присваивание
`window.__pinia = pinia`, что позволяет тестам читать живое состояние
(например, `liveData['БУР-12']['temp-engine'].value` для проверки факта
поступления данных от polling) и форсировать переходы состояний (например,
`connection.status = 'disconnected'` для эмуляции потери связи без
вмешательства в сетевые запросы). В production-сборке этот код
вырезается tree-shaking-ом Vite через проверку `import.meta.env.DEV`.

### Минимальные white-box-правки исходного кода

Для технологической поддержки тестируемости в продуктовый код внесены
шесть точечных изменений:

1. `data-testid` на пяти компонентах (см. выше).
2. Однострочный guard в `useThresholdMonitor.checkThresholds` —
   проверка флага `rgm:e2e:disable-thresholds`, при которой проверка
   порогов пропускается. Без этого флага случайные значения от polling
   могли бы триггерить критическое модальное окно посреди теста и
   блокировать клики.
3. Экспозиция `window.__pinia` в DEV-сборке.
4. Добавление в ESLint конфигурацию node-окружения для `playwright.config.js`,
   `vite.config.js` и `tests/e2e/**`.

Все правки задокументированы в коммитах с префиксом `test(e2e):` или
`chore(lint):` для трассируемости.

## 5.3 Тестовая стратегия: чек-лист, сценарии, классификация проверок

### Источник требований

Стратегия построена на основе технических требований, изложенных в
`docs/requirements.md`: тринадцать функциональных (ФТ-1…ФТ-13) и девять
нефункциональных (НФТ-1…НФТ-9). Каждое требование, поддающееся
автоматизированной верификации, получило хотя бы одну проверку.

Не покрыты автоматизированными e2e-тестами:
— **НФТ-1** (отклик ≤ 300 мс при 12 виджетах) и **НФТ-2** (загрузка ≤ 3 с,
бандл ≤ 1 МБ) — измеряются отдельными performance-инструментами вне
e2e-набора, поскольку в локальной dev-сборке Vite добавляет существенный
оверхед. Бюджет 1.5 секунды на отклик и 5 секунд на DOMContentLoaded в
DEV-режиме введён как proxy в P2-наборе.
— **НФТ-8** (расширяемость архитектуры) — структурное требование,
проверяется code-review, а не runtime-поведением.

### Жизненный цикл проектирования тестов

Перед написанием кода была пройдена цепочка:

1. **Анализ требований** — каждый ФТ/НФТ декомпозирован на проверяемые
утверждения. Результат зафиксирован в документе
`docs/superpowers/specs/2026-05-01-e2e-tests/01-requirements-analysis.md`.

2. **Чек-лист проверок** — плоский список булевых утверждений вида «при
действии X в состоянии Y наблюдается Z». Сформирован в файле
`02-checklist.md`. Объём — порядка 80 пунктов.

3. **Детализация проверок** — для каждого пункта чек-листа описаны
предусловие (Pre), действие (When), ожидание (Then) и побочный эффект на
состояние (`localStorage`). Файл `03-detailed-checks.md`.

4. **Пользовательские сценарии** — комбинации проверок, моделирующие
реальные рабочие смены трёх ролей. Сценарии S-E1..E4 для инженера,
S-M1..M3 для механика, S-F1..F2 для мастера. Файл `04-user-scenarios.md`.

5. **План расширения покрытия** — после реализации smoke-уровня проведён
gap-анализ и составлен план расширенных проверок с приоритизацией P0/P1/P2.
Файл `05-extended-coverage.md`.

### Классификация проверок

Все 137 тестов разделены на четыре категории по глубине проверки и
важности для приёмки:

**Smoke-проверки (минимально необходимые).** По одному happy-path тесту на
каждое функциональное и нефункциональное требование. Подтверждают, что
функция в принципе работает. Файлы `ft-NN-*.spec.js` и `nfr-NN-*.spec.js`.

**Critical path (расширенные позитивные пути).** Все ветки бизнес-логики:
переходы статусов наряда (`planned → in_progress → review → completed |
cancelled`), все шесть состояний шага наряда (`pending`, `in_progress`,
`passed`, `failed`, `skipped`), создание/удаление/отключение оборудования,
сохранение/сброс/восстановление раскладки панели мониторинга.

**Ролевые сценарии (интеграционные пользовательские пути).** Имитация
полной рабочей смены каждой из трёх ролей с переключением между разделами
системы. Например, S-F1: мастер открывает приложение, видит наряды на
приёмке, открывает один из них, утверждает и просматривает сгенерированный
акт.

**Edge cases и негативные проверки.** Прямой ввод URL на несуществующий
ID, повреждённый JSON в `localStorage`, отсутствие прав у роли, незаполненные
обязательные поля формы, поведение при потере связи, доступ механика к
чужому наряду в режиме «только чтение».

### Пример полного тест-кейса

Тест-кейс «Инженер создаёт наряд ТО-1 для БУР-12 и назначает его
на механика Петрова» (фрагмент `maintenance-create-extended.spec.js`):

```javascript
test('engineer: полный happy-path → наряд создан и виден в /maintenance', async ({
  page, loginAs,
}) => {
  await loginAs('engineer')
  await gotoCreate(page)
  await pickEquipmentSelect(page, 'БУР-12')
  await pickTypeSelect(page, '^ТО-1$')
  await page.getByRole('button', { name: 'Далее' }).click()
  await expect(page.getByRole('heading', { name: /Шаг 1/ }).first()).toBeVisible()
  await page.getByRole('button', { name: 'Далее' }).click()
  const mechanicSelect = page.getByRole('combobox')
    .filter({ hasText: /Выберите механика|Петров|Сидоров/ }).first()
  await mechanicSelect.click()
  await expect(page.getByRole('option').first()).toBeVisible({ timeout: 10_000 })
  await page.getByRole('option', { name: /Петров/ }).click()
  await page.getByRole('button', { name: /Создать наряд/ }).click()
  await expect(page).toHaveURL(/\/maintenance$/)
  await expect(page.getByText('БУР-12').first()).toBeVisible()
})
```

Тест-кейс описывает реальный сценарий рабочей смены: инженер открывает
форму создания, последовательно проходит три шага мастера, выбирает
исполнителя из списка механиков, нажимает «Создать наряд» и проверяет, что
после редиректа на список нарядов виден созданный наряд для БУР-12 с
исполнителем Петровым. На каждом шаге используется явная синхронизация
(`expect(...).toBeVisible()`) для устойчивости к асинхронной загрузке.

### Решение проблем устойчивости тестов

В процессе разработки были обнаружены и устранены пять характерных классов
проблем:

1. **Race-condition при чтении состояния, инициализированного в onMounted.**
Решение: ассерции через `expect.poll(() => ...)` с явным таймаутом вместо
`page.waitForTimeout`.

2. **Повторная очистка `localStorage` через `addInitScript` при reload.**
Решение: сторожевой флаг `__e2e_cleaned__` в `sessionStorage`, при наличии
которого скрипт пропускает очистку.

3. **Cyrillic в URL ломает прямой `page.goto('/equipment/БУР-12')`.**
Решение: навигация через клик по карточке списка, как делает реальный
пользователь.

4. **Критическое модальное окно перехватывает клики после первого
полного цикла polling.** Решение: e2e-флаг `rgm:e2e:disable-thresholds`,
выключающий пороговый монитор в тестовой среде.

5. **Auth-guard срабатывает быстрее, чем `fetchUser` восстанавливает
сессию.** Решение: ожидание `expect.poll(() => __pinia.auth.user?.role)`
перед навигацией на защищённые маршруты.

### Параллельный запуск и изоляция

Playwright по умолчанию запускает тесты в нескольких параллельных
воркерах. Каждый тест получает свежий браузерный контекст с собственным
`localStorage`, поэтому сидированные данные не перетекают между тестами.
В рамках одного спек-файла тесты независимы: фикстура `cleanState`
автоматически восстанавливает чистое состояние перед каждым из них.

3. Текстовые подписи кнопок в `StepCard` — «Выполнено», «Не выполнено»,
«Пропущено» (а не «С замечаниями»/«Пропустить», как было в плане). POM
обновлён, тесты приведены в соответствие.

## Приложение Г. Шаблоны абзацев для введения и заключения главы

### Введение в главу 5 (вставить перед 5.1)

Тестирование клиентской части системы мониторинга выполнено в виде
автоматизированного набора end-to-end проверок на основе фреймворка
Playwright. Выбор автоматизации в качестве основного подхода обусловлен
тем, что разработанная система содержит большое количество интерактивных
сценариев (конструктор панели мониторинга, мастер создания нарядов на
техническое обслуживание, пошаговое выполнение работ, формирование
актов), каждый из которых требует регулярного регрессионного контроля
при дальнейшем развитии проекта. Реализованный набор из 137 тест-функций
покрывает все 13 функциональных требований и 6 из 9 нефункциональных
требований из технического задания и проходит за 2 минуты с нулевым
количеством падений.

В разделе 5.1 обосновывается выбор методологии и инструментов. В разделе
5.2 описывается архитектура тестового набора — структура каталогов,
Page Object Model, фикстуры и приёмы изоляции состояния. В разделе 5.3
изложена тестовая стратегия и категоризация проверок (smoke, critical
path, ролевые сценарии, edge cases) с указанием соответствия требованиям.
Раздел 5.4 содержит примеры реализованных тест-кейсов и описание
итеративного процесса разработки тестов. Заключительный раздел 5.5
анализирует результаты прогона, метрики покрытия и выявленные в ходе
разработки несоответствия между техническим заданием и фактическим
поведением системы.

### Заключение по главе 5 (вставить после 5.5 в качестве итогового абзаца)

Разработанный набор автоматизированных e2e-тестов является не
формальным дополнением к коду, а полноценным верификационным механизмом,
интегрированным в процесс разработки. На каждое функциональное и
применимое нефункциональное требование из технического задания
сопоставлено как минимум одно проверочное утверждение, а критические
бизнес-сценарии (жизненный цикл наряда на ТО, конструктор панели
мониторинга, ролевая модель доступа) проверены на нескольких уровнях
глубины — от smoke-проверок до пошаговых интеграционных сценариев.
Полученные характеристики набора (137 тестов, 100 % успешного прохождения
в Chromium, длительность прогона 2.1 минуты, кросс-браузерная
совместимость с тремя движками) удовлетворяют практическим критериям
приёмки автоматизированного тестирования: набор достаточно полон для
обнаружения регрессий, достаточно быстр для запуска при каждой
сборке и достаточно структурирован для поддержки при эволюции
интерфейса.

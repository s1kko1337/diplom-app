# 4.4 Реализация frontend-части системы

Frontend-часть системы мониторинга бурового оборудования завода «Рудгормаш»
проектируется как основной слой визуализации и оперативного взаимодействия
пользователя с производственными процессами. На клиентской стороне
сосредоточена вся логика отображения данных в реальном времени, настройки
персональных рабочих мест, подготовки нарядов на техническое обслуживание,
формирования отчётов и анализа эффективности парка из восьми буровых станков.
Основной упор сделан на интерактивность, адаптивность под рабочие места
разного формата (от инженерной станции до планшета мастера в цеху) и
визуальную наглядность ключевых производственных метрик. Решение реализовано
на стеке Vue 3.5 (Composition API, `<script setup>`), Vite 7, Pinia 3,
Vue Router 5, Tailwind CSS 4 и ECharts 6 в связке с vue-echarts 8; для
конструктора панелей мониторинга используется библиотека grid-layout-plus,
для drag-and-drop виджетов — её собственный механизм сетки.

## Базовый каркас интерфейса

Каркас приложения построен по схеме «sidebar + content» и реализован в
компоненте `DefaultLayout`. В его состав входят боковая панель навигации
`AppSidebar`, шапка `AppHeader`, основная рабочая область `<RouterView />`,
подвал `AppFooter` и глобально подключённое модальное окно
`CriticalAlertModal`, которое всплывает поверх любой страницы при
поступлении критического уведомления. Для экрана авторизации применяется
отдельный лёгкий layout `AuthLayout` без навигации.

Боковая панель `AppSidebar` поддерживает два режима: на десктопе она
работает как сворачиваемая панель шириной 14/55 единиц (узкая по умолчанию,
расширяется по hover или закрепляется кнопкой «pin»); на мобильных
устройствах — как выезжающая панель `Sheet`, которую можно открыть
кнопкой-«гамбургером» в шапке. Состав пунктов меню фильтруется в
зависимости от роли текущего пользователя через утилиту `filterMenuByRole`,
поэтому механик, мастер и инженер видят разные наборы разделов. В нижней
части панели расположены переключатель темы (`ThemeToggle`) и меню
пользователя (`UserMenu`).

Шапка `AppHeader` включает «хлебные крошки» (`AppBreadcrumb`), индикатор
текущего времени (composable `useClock`), статус подключения к серверу
(`ConnectionStatus`), кнопку перехода в раздел уведомлений со счётчиком
неподтверждённых сообщений в виде красного бейджа и пользовательское меню.

## Компонент ConnectionStatus

Одним из ключевых элементов шапки является компонент `ConnectionStatus`,
визуализирующий состояние связи с back-end. Он принимает данные из Pinia-
стора `useConnectionStore` и поддерживает три состояния: «ПОДКЛЮЧЕНО»
(зелёная точка, нейтральная рамка), «ПЕРЕПОДКЛЮЧЕНИЕ…» (жёлтая
пульсирующая точка, жёлтая рамка) и «НЕТ СВЯЗИ» (красная пульсирующая
точка, красная рамка). Логика смены состояний построена на счётчике
последовательных неудач: одна ошибка опроса переводит систему в режим
переподключения, три и более — в состояние полного разрыва. При успешном
ответе счётчик сбрасывается, а в стор записывается метка
`lastSuccessfulUpdate` для целей диагностики. Стор обновляется из
композаблов опроса датчиков через методы `reportSuccess` и `reportFailure`,
что обеспечивает единый источник правды о состоянии связи на всех экранах.

## Тема оформления

Поддерживается две темы — светлая и тёмная. Они построены на CSS-
переменных, объявленных в `assets/theme.css`; переключение выполняется
добавлением класса `.dark` к корневому элементу документа. За управление
темой отвечает composable `useTheme`, который читает значение из
пользовательского стора предпочтений `usePreferencesStore`, синхронизирует
его с `localStorage` и пересчитывает классы документа через `watchEffect`.
Для графиков ECharts значения цветов из CSS-переменных дополнительно
проксируются composable-ом `useChartColors`, благодаря чему линии,
шкалы и подписи на графиках мгновенно перестраиваются при смене темы.

## Авторизация и выбор роли

Экран `LoginView` реализует упрощённую процедуру входа для целей демо-
версии: пользователь выбирает учётную запись из выпадающего списка
(`Select`) и вводит произвольный пароль. Список заполняется коллекцией
`MOCK_USERS`, в которой представлены четыре сотрудника с ролями
«инженер», «механик», «механик» и «мастер». При успешном входе токен
сохраняется в `localStorage`, выполняется однократная миграция legacy-
ключей mock-runtime (`migrateLegacyKeys`), загружаются персональные
настройки пользователя через `usePreferencesStore.load()`, после чего
происходит редирект на исходный маршрут (если он был сохранён в
`route.query.redirect`) или на главную страницу. В случае ошибки
авторизации сообщение отображается в красной плашке поверх формы.

## Главный экран и ролевые сценарии

Главный экран (`HomeView`) спроектирован как «обзор системы» с
адаптацией под роль авторизованного пользователя.

— Инженер видит полный набор панелей: общие KPI парка (`HomeMetricsPanel`),
сетку плиток оборудования (`HomeEquipmentGrid`), сводку по техническому
обслуживанию (`HomeMaintenanceSummary`) и две колонки внизу — ближайшие
ТО (`HomeUpcomingMaintenance`) и активные уведомления (`HomeAlertsPanel`).

— Мастер получает тот же набор сверху, но в нижнем блоке вместо
ближайших ТО показана очередь нарядов «На приёмке», сформированная
компонентом `HomeOrdersCard` в режиме `mode="review"`.

— Механик видит максимально сжатый интерфейс, ориентированный на
выполнение работы: только список назначенных на него активных нарядов
(`mode="assigned"`) и панель уведомлений.

При монтировании экрана параллельно через `Promise.all` инициируется
загрузка списка оборудования и нарядов ТО, затем для каждой единицы
дозагружаются её детали — это позволяет показать сводные индикаторы
без отдельных запросов из плиток.

## Список оборудования

Экран `EquipmentListView` отображает весь парк бурового оборудования.
В верхней части расположены заголовок с подзаголовком, карточка-счётчик
«АКТИВНО N/M» и кнопка «Подключить станок», открывающая диалог
`EquipmentCreateDialog`. Ниже находится строка поиска с иконкой `Search`
и группа вкладок `Tabs` для фильтрации по статусам: все, в работе,
простой, неисправность, отключено; в каждой вкладке отображается
актуальное количество единиц.

Сами станки представлены в двух взаимодополняющих видах. Сетка
карточек (`Card`) показывает идентификатор, бейдж статуса с цветной
точкой (`STATUS_DOT_COLORS`), модель, три ключевых датчика
(температура двигателя, обороты, глубина), время до ближайшего ТО,
а также кнопки управления — «Отключить» / «Подключить» и «Удалить»,
открывающие диалог подтверждения `EquipmentActionsDialog`. Под сеткой
карточек на десктопе расположена детальная таблица того же набора
данных, что удобно при сравнении большого числа станков. Все
датчики получают актуальные значения через `useSensorsStore.getSensorValue`
с откатом на последнее сохранённое значение из детали оборудования.

## Карточка станка с детализацией

Экран `EquipmentDetailView` представляет собой страницу-«паспорт»
конкретного станка. В заголовке (`EquipmentHeader`) выводятся
идентификатор, модель и кнопка управления опросом датчиков; справа
индикатор `LiveDataIndicator` показывает время с момента последнего
обновления. Контент разделён на пять вкладок:

1. «Обзор» — сводные метрики (`EquipmentMetrics`), полосы здоровья
   подсистем (`EquipmentStatusBars`) и линейные графики ключевых
   датчиков (`EquipmentCharts`).
2. «Датчики» — таблица всех 12 датчиков с текущими значениями,
   единицами измерения и пороговыми отметками (`EquipmentSensors`).
3. «ТО» — вкладка `MaintenanceTab`, объединяющая историю выполненных
   обслуживаний и активные наряды по этому станку.
4. «Детали» — история замен запчастей с артикулами и стоимостью
   (`EquipmentParts`).
5. «История» — обозреватель архивных значений датчиков
   (`SensorHistoryExplorer`) и блок технических характеристик
   (`EquipmentSpecs`).

При входе на страницу запускается polling живых данных с интервалом
5 секунд через `useSensorsStore.startPolling`; при размонтировании
опрос корректно останавливается.

## Конструктор панелей мониторинга

Центральным элементом инженерного рабочего места является
конструктор персональных панелей `EquipmentDashboardView`. Он
позволяет пользователю собрать индивидуальную раскладку из набора
виджетов под конкретный станок и сохранить её в пользовательской
коллекции `dashboards` mock-runtime (`scope: 'user'`).

Раскладка реализована поверх библиотеки `grid-layout-plus`
(`GridLayout` + `GridItem`). Сетка использует 12 колонок на
десктопе и 1 колонку на мобильных устройствах с автоматической
вертикальной перекомпоновкой при пересечении breakpoint. Важной
архитектурной особенностью является то, что массив раскладки
обязан быть мутабельным `ref`, а не `computed`: библиотека меняет
элементы in-place при перетаскивании и изменении размеров,
поэтому `computed` приводит к разрыву реактивности (это
зафиксировано в правилах проекта). Все изменения сетки
отправляются в стор `useDashboardsStore` через обработчик
`@layout-updated`, который только записывает координаты, не
форсируя пересоздание массива.

Конструктор имеет два режима — просмотр и редактирование. В
режиме просмотра виджеты статичны и оптимизированы для чтения. В
режиме редактирования (`dashboardsStore.editing === true`)
включаются drag, resize и кнопка удаления виджета (крестик в
правом верхнем углу `WidgetWrapper`); вокруг каждого виджета
появляется тонкая рамка `ring-1 ring-primary/30`, подсказывающая
интерактивность. Кнопка «Настроить» переключает режим, после
выхода из режима автоматически вызывается `saveConfig`. Кнопка
«Сброс» восстанавливает дефолтную раскладку через `resetConfig`,
а кнопка «Виджет» открывает модальное окно `AddWidgetModal`.

## Реестр виджетов и WidgetWrapper

Все доступные виджеты описаны в реестре `widgetRegistry.js`,
оформленном как объектная карта `widgetTypes`. Каждый тип
содержит компонент (загружаемый лениво через
`defineAsyncComponent`), человекочитаемое название, описание,
размер по умолчанию `{ w, h }` и флаг `requiresSensor`,
определяющий, нужен ли виджету привязанный датчик. На текущий
момент в системе реализовано одиннадцать типов виджетов,
сгруппированных в две категории.

Сенсорные виджеты (`requiresSensor: true`):

— `numeric-indicator` — крупное числовое значение датчика с
единицей измерения и индикатором попадания в порог
(норма/внимание/критическое);

— `line-chart` — линейный график изменения значения во времени
с автоматической генерацией демо-данных, если история ещё не
накоплена;

— `gauge` — круговая шкала с цветовой индикацией порогов,
строящаяся на ECharts `GaugeChart`;

— `sensor-history` — график за последние 24 часа по выбранному
датчику.

Информационные виджеты (`requiresSensor: false`):

— `info` — модель, серийный номер, год выпуска;

— `status` — крупный визуальный индикатор текущего состояния
оборудования;

— `specs` — таблица технических параметров;

— `service-history` — записи о проведённых ТО и ремонтах;

— `parts` — история замены запчастей;

— `maintenance-timeline` — таймлайн циклов технического
обслуживания;

— `subsystem-health` — состояние подсистем оборудования.

Универсальная обёртка `WidgetWrapper` резолвит компонент по
типу через `getWidgetComponent`, передаёт ему `value`
(текущее значение датчика, прочитанное из `useSensorsStore`),
определение датчика `sensor`, объект оборудования и его
идентификатор. Если тип виджета неизвестен (например, после
отката версии конфигурации), отображается заглушка
«НЕИЗВЕСТНЫЙ ВИДЖЕТ».

Модальное окно `AddWidgetModal` адаптивно: на десктопе
открывается как `Dialog`, на мобильных — как `Sheet` с
выездом снизу. Внутри располагаются две секции — выбор типа
виджета и (если он того требует) выбор датчика; кнопка
«ДОБАВИТЬ» становится активной только после выполнения
обоих условий.

## Опрос данных и пороговый мониторинг

Опрос живых значений датчиков построен на стороне фронта
поверх единственного интервала `setInterval` в стора
`useSensorsStore`. Метод `startPolling(equipmentId, interval)`
запускает первичный вызов `fetchLive` и затем периодически
повторяет его с интервалом по умолчанию 5 секунд; при
размонтировании страницы `stopPolling` корректно очищает
таймер. Каждый успешный ответ обновляет состояние подключения
через `connectionStore.reportSuccess`, каждая ошибка — через
`reportFailure`, что напрямую отражается в индикаторе
`ConnectionStatus`.

После каждого успешного опроса автоматически вызывается
проверка порогов через composable `useThresholdMonitor`. Он
проходит по всем датчикам станка и сравнивает текущее
значение с пороговыми значениями `warning` и `critical`,
заданными в определении датчика. Чтобы избежать «штормa»
дублирующих уведомлений, в модульной переменной
`activeViolations` (Map) хранится текущий уровень нарушения
по каждому ключу `equipmentId:sensorId`. Уведомление
генерируется только при смене уровня — например, при
переходе из «нормы» в «warning» или из «warning» в
«critical»; при возврате в норму запись удаляется.
Сгенерированное уведомление попадает в стор
`useAlertsStore.addLiveAlert`, откуда оно рассылается
по всем подписанным компонентам — от бейджа в шапке до
панели на главной странице.

## Уведомления и критическое окно

Полный список уведомлений выведен на странице `AlertsView`.
В её верхней части показаны общий счётчик активных уведомлений
и пульсирующая красная плашка с числом критических, если
такие есть. Ниже расположен набор вкладок `Tabs` для
фильтрации по типу — критические, предупреждения,
обслуживание, информационные, неподтверждённые — каждая со
своим счётчиком. Сами уведомления группируются по дате
(«Сегодня», «Вчера», конкретные даты в русской локали) и
отображаются как карточки с цветной рамкой по типу. Каждая
карточка содержит иконку (`AlertTriangle`, `AlertCircle`,
`Info`, `Wrench`, `CheckCircle` из `lucide-vue-next`),
заголовок, описание, бейджи с идентификатором станка и
типом уведомления, кнопку «Подтвердить» и крестик для
удаления; подтверждение помечает запись бейджем
«Подтверждено» и снижает её визуальный вес (`opacity-60`).
Внизу страницы выведена статистическая панель —
пять карточек с разбивкой по типам.

Появление и исчезновение элементов оформлено через
Vue-`<Transition>` с эффектом `fade`. Подтверждение
конкретной записи в сторе выполняется заменой объекта
целиком (`alerts.value[idx] = { ...old, acknowledged: true }`) —
это особенность реактивности Pinia в проекте, без неё ряд
вычисляемых селекторов перестаёт пересчитываться.

Параллельно с обычными уведомлениями стор отслеживает
события типа `critical`. При их появлении на любом экране
поверх контента всплывает компонент `CriticalAlertModal` —
крупное модальное окно с пульсирующим красным фоном,
блокирующее интерфейс до подтверждения. Окно построено
адаптивно: `Sheet` снизу на мобильных и `Dialog` на
десктопе. В нём выведены идентификатор оборудования,
заголовок и описание проблемы, время события, а также две
кнопки — «ПОДТВЕРДИТЬ И ЗАКРЫТЬ» и заглушка «ЭКСТРЕННАЯ

## Управление техническим обслуживанием

Раздел «ТО» объединяет несколько связанных рабочих процессов и
оформлен в виде четырёх экранов: список нарядов
(`MaintenanceView`), создание наряда (`MaintenanceCreateView`),
детальная карточка наряда с пошаговым выполнением
(`MaintenanceDetailView`) и просмотр официального документа —
акта или ЕО (`MaintenanceDocumentView`).

### Канбан-доска нарядов

Список нарядов выведен на доску `KanbanBoard`, где колонки
соответствуют статусам жизненного цикла: «Запланировано»,
«В работе», «На приёмке», «Завершено» и (опционально)
«Отменено». Каждая колонка `KanbanColumn` цветовая, имеет
свой акцент (`bg-status-info`, `bg-status-warning`,
`bg-status-maintenance`, `bg-status-success` и серый для
отменённых) и собственный лимит видимых карточек. Сама
карточка наряда `OrderCard` показывает тип ТО, идентификатор,
оборудование, исполнителя, дату создания и прогресс-бар
выполнения шагов; для отменённых нарядов добавлена кнопка
«Восстановить» (видна только пользователям с правом
создания, проверяемым composable-ом `usePermissions`).

### Мастер создания наряда

Создание наряда оформлено как трёхшаговый мастер. На первом
шаге `CreateStepEquipment` выбирается единица оборудования и
тип обслуживания (ежедневное, ТО-1, ТО-2, ремонт). На втором
шаге `CreateStepChecklist` подгружается типовой чек-лист
для выбранного типа ТО, который мастер может донастроить —
добавить, удалить, переупорядочить пункты. На третьем шаге
`CreateStepAssign` выбирается исполнитель из списка
механиков, полученного через `getUsersByRole('mechanic')`,
и плановая дата выполнения. В верхней части экрана
расположен индикатор прогресса с тремя круглыми маркерами
и подписями шагов, кнопки «Назад/Далее» становятся активны
только после прохождения валидации текущего шага
(`isCurrentStepValid`). Доступ к экрану ограничен:
пользователи без права создания получают плашку «Недостаточно
прав» со ссылкой возврата к списку.

### Пошаговое выполнение ТО

Внутри карточки наряда механик работает с компонентом
`StepWizard`. В верхней части показан прогресс-бар
`StepProgress` с кружками-«кликабельными» шагами; при
переходе между шагами текущий пункт автоматически
переводится в статус `in_progress` через
`maintenanceStore.startStep`. Активный шаг отображается в
карточке `StepCard` с описанием работ, нормативами,
полями ввода фактических измерений (`MeasurementInput`),
расхода материалов (`MaterialInput`), таймером
(`StepTimer`) и кнопками завершения «Выполнено / С
замечаниями / Пропустить». После завершения всех шагов
интерфейс переключается в режим «Сводка» (`OrderSummary`),
где механик может отправить наряд на приёмку мастеру через
кнопку, эмитящую `submit-for-review`.

## Отчёты

Раздел отчётов поддерживает четыре типа документов:
«Акт выполнения ТО» (`maintenance_completion`),
«Отчёт об инциденте» (`incident_report`), «Отчёт смены»
(`shift_report`) и «Аналитическая сводка»
(`analytics_summary`). Список отчётов на странице
`ReportsView` представлен карточками с типом, заголовком,
кратким резюме, бейджем статуса (черновик / опубликован) и
датой создания; над списком расположены два фильтра
(`Select` по типу и по статусу) и кнопка «Сбросить».
Кнопка «Создать отчёт» видна только при наличии
соответствующих прав, проверяемых утилитой
`canCreateAnyReport`.

Создание отчётов реализовано на едином экране
`ReportCreateView`, который динамически подключает один из
четырёх компонентов-форм: `IncidentForm`, `ShiftForm`,
`AnalyticsForm` либо тип «акт ТО», создаваемый
автоматически по завершении наряда. Просмотр отчёта на
экране `ReportDetailView` подбирает соответствующий
компонент-payload (`IncidentPayload`, `ShiftPayload`,
`AnalyticsPayload`, `MaintenancePayload`), который
типизированно рендерит структурированные данные.

## Аналитика

Экран `AnalyticsView` строится из нескольких независимых
блоков: переключатель периода (`AnalyticsPeriodSelector` —
неделя, месяц, 3/6 месяцев, год), карточки KPI по парку
(`AnalyticsKPICards`), набор линейных и столбчатых
графиков по выбранному периоду (`AnalyticsCharts`),
таблица эффективности по каждой единице оборудования
(`EfficiencyTable`) и статистика по ТО (`MaintenanceStats`).
Все графики строятся на ECharts через vue-echarts и
автоматически перекрашиваются при смене темы благодаря
`useChartColors`.

## Настройки

Экран `SettingsView` объединяет пользовательские и
системные настройки в пять вкладок: «Отображение» (тема,
плотность интерфейса, локаль), «Уведомления» (правила
показа и звуковые оповещения), «Пороги» (редактирование
пороговых значений датчиков для всего парка),
«Безопасность» (смена пароля, политика сессий), «Система»
(обслуживание mock-runtime — кнопки «Сбросить демо»,
сведения о версии). Каждая вкладка реализована отдельным
компонентом, который через `defineExpose` предоставляет
методы `save()` и `reset()`; верхнеуровневые кнопки
«СОХРАНИТЬ» и «СБРОСИТЬ» в шапке экрана последовательно
вызывают эти методы у всех вкладок и показывают
временное сообщение об успехе.

## Архитектурные особенности и стили

Все перечисленные компоненты выполнены с использованием
Tailwind CSS 4 (utility-first), без `inline`-стилей; цвета,
типографика и отступы заданы через токены и CSS-переменные
из `assets/theme.css`, что обеспечивает единое визуальное
оформление и мгновенное переключение между светлой и тёмной
темами. Шрифты — Inter (основной) и JetBrains Mono (для
числовых метрик `.metric-value`).

Состояние приложения распределено между девятью Pinia-
сторами (composition syntax): `auth`, `equipment`, `sensors`,
`alerts`, `dashboards`, `maintenance`, `reports`,
`preferences`, `connection`. Pinia-сторы дополняются
специализированными composables (`useClock`,
`useBreakpoint`, `useSwipe`, `useTheme`, `useChartColors`,
`useChartOptions`, `useLiveData`, `usePermissions`,
`useThresholdMonitor`), которые инкапсулируют побочные
эффекты и обеспечивают повторное использование логики между
экранами.

Все маршруты определены в `Vue Router 5` с lazy-load
импортом представлений и парой guard-ов: `requiresAuth`
выполняет редирект неавторизованных пользователей на
`/login`, `guest` — наоборот, не пускает уже авторизованных
на форму входа. Имена маршрутов записаны в kebab-case,
алиас `@` указывает на каталог `src/`.

Интерфейс полностью адаптивен: все ключевые экраны
проверены на трёх breakpoint-ах (мобильный, планшет,
десктоп), сложные модальные окна реализованы через пары
`Dialog`/`Sheet`, элементы управления имеют минимальный
размер тач-цели 44 × 44 px на мобильных устройствах, а
сетка панели мониторинга автоматически переключается в
одну колонку при ширине экрана меньше планшетной.

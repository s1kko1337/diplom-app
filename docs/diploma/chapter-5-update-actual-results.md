# Корректировки главы 5 «Тестирование» по итогам фактического прогона

> Документ содержит правки и дополнения к существующему тексту главы 5
> (`chapter-5-testing-materials.md`) на основе двух фактических прогонов:
> локального в Chromium и полного кросс-браузерного. Ниже — точные цифры,
> готовые абзацы для замены и описание выявленных кросс-браузерных
> особенностей с уже применёнными правками.

## 1. Сводка двух прогонов (для замены текущей сводки в 5.4)

### Прогон 1. Локальный в Chromium

```
$ npm run test:e2e -- --project=chromium

  140 passed
    4 skipped
    0 failed
  Duration: 2.1 minutes
```

Этот прогон выступает основным для повседневной разработки и для
встраивания в процесс непрерывной интеграции. Длительность 2,1 минуты
позволяет запускать его при каждом push'е в ветку и при подготовке
pull request'ов.

### Прогон 2. Полный кросс-браузерный (Chromium + Firefox + WebKit)

```
$ npm run test:e2e

  Running 432 tests using 3 workers
  …
  418 passed
   12 skipped
    2 failed
  Duration: 9.4 minutes
```

Расшифровка:

— **432 ассерт-кейса всего** — 144 тестовых сценария умножаются на три
браузерных проекта; небольшое отклонение от арифметики (144 × 3 = 432)
объясняется параметризацией адаптивных тестов по четырём контрольным
точкам экрана.

— **12 пропущенных** = 4 теста с `test.skip` × 3 браузера. Все четыре
пропущенных теста (два на критическое модальное окно и два на сценарии
выхода из системы) согласованно пропускаются во всех трёх движках.

— **2 упавших** — кросс-браузерные особенности, выявленные именно
полным прогоном (см. раздел 3 ниже): один сбой в Firefox, один в WebKit.
В Chromium те же тесты проходят без ошибок.

### Сравнительная сводка

| Параметр | Chromium-only | Cross-browser |
|----------|---------------|---------------|
| Тестовых ассерт-кейсов | 144 | 432 |
| Прошли успешно | 140 | 418 |
| Пропущены явно (`test.skip`) | 4 | 12 |
| Завершились с ошибкой | 0 | 2 (1 в Firefox, 1 в WebKit) |
| Длительность | 2,1 мин | 9,4 мин |
| Параллельных воркеров | 3 | 3 |

## 3. Новый подраздел в 5.4 — «Кросс-браузерные особенности и устранение»

Этот подраздел рекомендуется вставить после описания сводных метрик
(перед таблицей соответствия требований).

### Кросс-браузерные особенности и их устранение

Полный прогон в трёх движках выявил два расхождения, не воспроизводимых
в Chromium и связанных с особенностями реализации спецификаций
браузерами Firefox и WebKit. Оба случая были разрешены без отказа от
проверок и без введения условной логики «`if (browserName === 'firefox')
…`», что подтверждает применимость единого тестового набора для всех
поддерживаемых браузеров.

**Случай 1. Firefox: проверка focus-visible после первого нажатия Tab.**
В тесте `focus visible на интерактивах: первый Tab не теряет фокус`
проверяется, что после раскрытия бокового меню и одного нажатия клавиши
Tab активный элемент документа отличен от элемента BODY. В движках
Chromium и WebKit указанный сценарий стабилен. В Firefox начальная точка
фокуса документа может оставаться на корневом элементе HTML или BODY до
тех пор, пока пользователь явно не взаимодействует со страницей нажатием
мыши или другим стартовым событием. Тест переработан в две стороны:
вначале выполняется явный программный клик по элементу `document.body`
для установки детерминированной точки старта, затем последовательно
выполняется до пяти нажатий Tab с проверкой того, что хотя бы одно из них
переводит фокус на интерактивный элемент (BUTTON, INPUT, A, SELECT или
TEXTAREA). Подобная формулировка эквивалентна исходному критерию
доступности интерфейса, но не зависит от стартового состояния фокуса
браузера.

**Случай 2. WebKit: задержка применения CSS-класса при переключении
темы.** В тесте `таб «Отображение»: переключение светлой темы убирает
класс .dark` после двух последовательных нажатий кнопок «ТЁМНАЯ» и
«СВЕТЛАЯ» проверяется отсутствие класса `dark` на корневом элементе
документа. В Chromium и Firefox реактивность Pinia-стора предпочтений и
зависящего от него `useTheme` через `watchEffect` срабатывает в пределах
одного микро-таска. В WebKit между моментом завершения обработчика клика
и моментом применения нового значения класса наблюдается ненулевая
задержка, при которой синхронное чтение `document.documentElement.
classList` всё ещё возвращает предыдущее значение `dark`. Тест переработан
с использованием `expect.poll(...)`, который повторно опрашивает значение
с интервалом до семи секунд (значение по умолчанию для ассерций
Playwright). После замены синхронной проверки на `expect.poll` тест
устойчиво проходит во всех трёх движках без увеличения времени прогона
для Chromium и Firefox.

Применённые правки также повлекли мелкое улучшение конфигурации
тестового набора: ранее npm-скрипт `test:e2e:report` пытался открыть
HTML-отчёт по умолчанию из каталога `playwright-report/`, тогда как
конфигурация Playwright помещает отчёт в `tests/e2e/.report/`. Скрипт
скорректирован на явный путь, что устраняет расхождение между
ожидаемым и фактическим расположением отчёта.

## 5. Корректировка таблицы покрытия требований (для приложения В)

Поскольку полный прогон подтвердил выполнение требования НФТ-9
(кросс-браузерность) на практике, а не только конфигурационно,
рекомендуется в финальной таблице покрытия добавить столбец «Подтверждено
прогоном»:

| Требование | Тестов | Подтверждено прогоном |

| ФТ-1..ФТ-13 | 137 | да, в Chromium |
| НФТ-3 (адаптив 768/1440/1920) | 4 (параметризованные) | да |
| НФТ-4 (touch ≥ 44 px) | 1 | да |
| НФТ-5 (last data on offline) | 1 | да |
| НФТ-6 (recovery без reload) | 1 | да |

Под примечанием к строке НФТ-9 стоит пометить, что Edge и Chrome
основаны на одном и том же движке Chromium, а WebKit обеспечивает
проверку поведения на Safari-стеке (близко к Edge на iOS), что в
совокупности перекрывает требование «две последние стабильные версии
Google Chrome, Mozilla Firefox и Microsoft Edge».

## 6. Дополнение к разделу про найденные несоответствия план/код

Текущий текст перечисляет три таких расхождения. После полного прогона
выявлены ещё два, которые имеет смысл добавить в главу:

**Несоответствие 4.** Конфигурация HTML-отчёта Playwright в файле
`playwright.config.js` указывает каталог `tests/e2e/.report` через опцию
`outputFolder`, тогда как утилита `playwright show-report` без аргументов
ищет отчёт в стандартном каталоге `playwright-report`. Обнаружено в
момент попытки открыть отчёт сразу после прогона; устранено явной
передачей пути в `npm run test:e2e:report`.

**Несоответствие 5.** В Firefox начальный фокус документа после открытия
страницы остаётся на корневом элементе до первого пользовательского
взаимодействия, что приводит к необходимости явной инициации фокуса в
тестах доступности — особенность, не описанная в техническом задании,
но существенная для устойчивости проверок раздела `a11y-perf-smoke`.

# Расширенное покрытие e2e (P0/P1/P2)

## Текущее состояние (53 теста)

Покрыты ФТ-1..ФТ-13 и НФТ-3/4/5/6/7/9 в формате smoke + ключевые happy-path + role-сценарии. Каждое требование имеет минимум один проверочный тест.

## Принцип расширения

Smoke ≠ полное покрытие. Для каждой пользовательской функции должны быть протестированы:

1. **Happy path** — основной успешный сценарий (уже есть).
2. **Critical path** — все ветки бизнес-логики, переходы состояний, валидации.
3. **Edge cases** — невалидные входы, граничные значения, отсутствие данных, конкурентность.
4. **Error handling** — поведение при сбоях, восстановление.
5. **Role variants** — что доступно/недоступно каждой роли в каждом сценарии.

## Gap-анализ по доменам

### Auth (gap: высокий)
**Покрыто:** mock-инжекция токена, ролевая фильтрация меню, login через UI (1 smoke).
**Не покрыто:**
- Валидация формы логина (пустой пароль, ввод-без-пользователя).
- Logout: кнопка в UserMenu, очистка `localStorage`, редирект на `/login`.
- Auth guard: незалогиненный → `/login` с `?redirect=...`.
- Guest guard: залогиненный пытается зайти на `/login` → редирект на `/`.
- Восстановление сессии после reload (`fetchUser` + `getMe`).
- Race-condition: переход на защищённый маршрут до завершения `fetchUser`.

### Equipment (gap: средний)
**Покрыто:** список (фильтры, поиск), детальная карточка с табами, parts table, sensor history, удаление (только подтверждение).
**Не покрыто:**
- `EquipmentCreateDialog`: форма создания со всеми полями, валидация ID/модели.
- Удаление: подтверждение → реальное исчезновение карточки + декремент счётчика.
- Отключение/подключение: переходы статуса в обоих направлениях.
- Сортировка/группировка в таблице (если есть).
- Empty state при пустом списке.
- Mobile: таблица скрыта, видны только карточки.

### Dashboard (gap: высокий)
**Покрыто:** add/remove виджета, save/reload/reset, layout в localStorage, 3 типа виджетов (numeric/line/gauge).
**Не покрыто:**
- Остальные 8 типов виджетов: sensor-history, info, status, specs, service-history, parts, maintenance-timeline, subsystem-health.
- Drag-and-drop перемещение с проверкой нового layout.
- Resize виджета.
- AddWidgetModal: запрет добавления sensor-виджета без выбора датчика, кнопка ОТМЕНА.
- Mobile: одна колонка, viewport-переключение.
- Удаление всех виджетов → empty state «Нет виджетов».
- Editing toggle: переход в/из режима редактирования.

### Maintenance create (gap: высокий)
**Покрыто:** только smoke — engineer проходит 3 шага.
**Не покрыто:**
- Step 1 валидация: `Далее` неактивна без выбора оборудования и типа.
- Step 2: разные типы ТО (ЕО, ТО-1, ТО-2, ТО-3, ТР-1) → разные чек-листы.
- Step 2: добавление/удаление пунктов чек-листа.
- Step 3: валидация назначения (пустой исполнитель → кнопка неактивна).
- Назад/далее по шагам, сохранение состояния.
- Permission denied для mechanic/foreman: «Недостаточно прав».
- Cancel mid-creation (через breadcrumb или нав).

### Maintenance execute (gap: высокий)
**Покрыто:** start (planned→in_progress), 1 шаг passed, submit → review.
**Не покрыто:**
- Многошаговые наряды (5+ шагов): прогресс-бар, навигация.
- Пропущенный шаг с комментарием.
- Failed шаг с обязательным комментарием.
- Назад к завершённому шагу (просмотр).
- Cancel order (если разрешено инженеру).
- Materials: добавление, изменение количества.
- Step timer: запуск, остановка, фиксация времени.
- Order не для меня: mechanic2 видит чужой наряд → readonly.

### Maintenance review (gap: средний)
**Покрыто:** approve, return → in_progress.
**Не покрыто:**
- Return без причины: кнопка «Отправить» неактивна.
- Подсчёт пройденных/неуспешных шагов в OrderSummary.
- Просмотр акта (предварительный) до approve.
- Foreman: нет approve-кнопки на не-review нарядах.
- Engineer/mechanic: видит наряд в review только в readonly.

### Reports (gap: высокий)
**Покрыто:** list-page, RBAC (mechanic-доступ, /reports/create редирект, кнопка create).
**Не покрыто:**
- Создание incident_report от mechanic: форма + валидация + публикация.
- Создание shift_report от engineer: специфичные поля.
- Создание analytics_summary от engineer: период, агрегация.
- Создание maintenance_completion: автогенерация при approve наряда.
- Просмотр отчёта: типизированный payload рендерится корректно.
- Edit draft → publish.
- Фильтры: тип, статус, сброс.
- RBAC при создании: mechanic видит только incident в селекторе типа.

### Alerts (gap: средний)
**Покрыто:** seed visibility, badge, ack.
**Не покрыто:**
- Фильтр по типу: critical/warning/info/maintenance/unacknowledged.
- Группировка по дате (Сегодня/Вчера/конкретная дата).
- Bulk-операции (если есть).
- Empty state по фильтру.
- Crit alert модал: открывается, кнопка «Подтвердить и закрыть», блокирует UI.
- Live-генерация алертов от threshold-monitor (включая порог).

### Journal (gap: высокий)
**Покрыто:** smoke (таблица видна).
**Не покрыто:**
- Содержимое строк (тип, оборудование, исполнитель, дата).
- Фильтры по типу/оборудованию/дате/статусу.
- Клик по строке → переход на наряд / документ.
- Сортировка по дате.
- Empty state.

### Settings (gap: высокий)
**Покрыто:** save toast.
**Не покрыто:**
- Display: тема светлая/тёмная — переключение и применение.
- Notifications: правила и звуки.
- Thresholds: редактирование пороговых значений датчиков.
- Security: смена пароля.
- System: «Сбросить демо» → очистка `rgm:*` ключей.
- `?demo-reset=1` query.
- Reset all → проверка восстановления значений.

### Connection (gap: низкий)
**Покрыто:** 3 состояния, восстановление.
**Не покрыто:**
- Реальный сценарий: интервал между опросами и переход reconnecting→connected при возврате.
- LiveDataIndicator: время последнего обновления.
- Метка «Время последнего обновления» в шапке.

### Edge cases / error handling (gap: высокий)
**Не покрыто:**
- 404 / catch-all маршрут.
- Прямой URL на несуществующее оборудование (`/equipment/БУР-999`).
- Прямой URL на несуществующий наряд / отчёт.
- LocalStorage: corrupted JSON → авто-reseed.
- Demo reset: `?demo-reset=1` приводит к выходу + сбросу.
- Concurrent edits (если поддерживается).

### Accessibility / визуал (gap: средний)
**Не покрыто:**
- Keyboard navigation (Tab порядок).
- Focus visible на интерактивах.
- ARIA-labels на icon-only кнопках.
- Breadcrumbs корректно отражают путь.
- Print стили: `@media print` скрывает навигацию.
- Темы: dark↔light без артефактов.

## Стратегия исполнения

1. Каждый батч — отдельный subagent с фокусом на домене.
2. Перед запуском теста подтянуть актуальный UI через инспекцию (компонент → реальные тексты/роли).
3. Использовать существующие хелперы (`makeAlert`, `makeOrder`, `seedX`); расширять при необходимости.
4. Один спек-файл на батч, ≥ 5 тестов в нём.
5. Прогон в chromium → коммит.
6. Прагматичные `test.skip` с TODO для реально нерабочих сценариев (UI ещё не имеет нужного элемента).

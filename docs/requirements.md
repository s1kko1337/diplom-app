Технические требования

Технические требования к клиентской части информационной системы охватывают как функциональные аспекты – определяющие что система должна делать, так и нефункциональные – описывающие как она должна это делать. Совокупность данных требований формирует рамки реализации и критерии приёмки разработанного программного обеспечения.

Сводный перечень функциональных требований к клиентской части системы:

­ ФТ-1. Отображение перечня оборудования – вывод списка подключённого оборудования с индикацией статуса (в работе, простой, неисправность, отключено), детальная карточка при выборе.

­ ФТ-2. Настраиваемая информационная панель – добавление, удаление, перемещение и изменение размера виджетов на сеточной панели с сохранением конфигурации на сервере.

­ ФТ-3. Виджеты метрик – числовой индикатор, линейный график, шкальный индикатор с цветовой индикацией зон норма / предупреждение / критическое.

­ ФТ-4. Уведомления о пороговых значениях – визуальные оповещения при превышении порогов с указанием оборудования, датчика, значения и времени.

­ ФТ-5. Информирование о заменах деталей – отображение сведений о заменах комплектующих с выделением неоригинальных деталей.

­ ФТ-6. История показаний – просмотр данных датчиков за выбранный период с фильтрацией по оборудованию и параметрам.

­ ФТ-7. Получение данных в реальном времени – периодический опрос REST API с обработкой таймаутов, разрывов и ошибок сервера.

­ ФТ-8. Индикация статуса подключения – постоянный индикатор состояния связи: подключено / переподключение / нет связи.

­ ФТ-9. Управление нарядами на ТО – создание нарядов по шаблонам, жизненный цикл (создан → в работе → на проверке → выполнен / отклонён / отменён), пошаговое выполнение.

­ ФТ-10. Фиксация результатов работ – внесение измерений с проверкой соответствия нормативам, регистрация расхода материалов.

­ ФТ-11. Формирование документов – автоматическое создание актов выполненных работ в форматах HTML и JSON.

­ ФТ-12. Журнал технического состояния – ведение электронного журнала с привязкой записей к нарядам, фильтрация по критериям.

­ ФТ-13. Ролевое разграничение доступа – различные права для механиков, мастеров участков и диспетчеров.

В части функциональных возможностей система должна обеспечивать отображение перечня подключённого бурового оборудования с индикацией текущего статуса каждой единицы. Поддерживаемые статусы – «в работе», «простой», «неисправность» и «отключено» – должны визуально различаться посредством цветовой индикации. При выборе единицы оборудования из списка должна открываться детальная карточка, содержащая основные технические характеристики и историю последних событий. Обновление статусов должно происходить автоматически в соответствии с механизмом периодического опроса серверной части.

Ключевым функциональным элементом является настраиваемая информационная панель, на которой пользователь может размещать виджеты трёх типов: числовой индикатор для отображения текущего значения метрики с указанием наименования параметра и единицы измерения, линейный график для визуализации динамики изменения параметра во времени с поддержкой выбора временного интервала, а также шкальный индикатор, отображающий текущее значение на шкале с цветовой индикацией зон нормы, предупреждения и критического состояния. Каждый виджет привязывается к конкретному датчику конкретной единицы оборудования. Расположение виджетов организовано по сеточной модели с возможностью добавления, удаления, перемещения и изменения размера методом перетаскивания. Текущая конфигурация панели сохраняется на стороне сервера и восстанавливается при повторном входе пользователя в систему.

Система должна реализовывать механизм уведомлений, срабатывающий при достижении метриками датчиков заданных пороговых значений. Каждое оповещение должно содержать наименование единицы оборудования, наименование датчика, текущее и пороговое значения, а также временную метку события. Оповещение отображается до тех пор, пока значение метрики не вернётся в допустимый диапазон либо оператор не подтвердит его просмотр вручную.

Функция информирования о заменах деталей обеспечивает отображение сведений о произведённых заменах комплектующих для каждой единицы оборудования с указанием наименования детали, даты замены и признака оригинальности установленного комплектующего. Записи об установке неоригинальных деталей должны быть визуально выделены в интерфейсе. Предусмотрена также возможность просмотра исторических данных показаний датчиков за выбранный временной период с фильтрацией по оборудованию, конкретному датчику или группе параметров. Результаты отображаются в виде линейного графика и табличного представления.

Дополнительно система должна обеспечивать создание нарядов на техническое обслуживание на основе регламентных шаблонов чек-листов для различных типов работ (ТО-1, ТО-2, ТО-3, ежесменное обслуживание, текущий ремонт). Управление жизненным циклом наряда предполагает поддержку статусов «создан», «в работе», «на проверке», «выполнен», «отклонён» и «отменён» с контролем допустимости переходов между ними. Пошаговое выполнение работ включает фиксацию времени начала и завершения каждого этапа, внесение фактических результатов измерений с автоматической проверкой соответствия нормативным требованиям, а также регистрацию использованных материалов с указанием объёма и марки. По завершении наряда система обеспечивает автоматическое формирование акта выполненных работ в структурированном формате. Ведение электронного журнала технического состояния оборудования предусматривает привязку записей к нарядам, фильтрацию и поиск по множественным критериям. Ролевое разграничение доступа обеспечивает различные права для механиков, мастеров участков и диспетчеров.

Основным протоколом взаимодействия клиентской части с серверной частью является REST API. Для поддержки актуализации данных в режиме, приближённом к реальному времени, клиентская часть реализует механизм периодического опроса сервера. Клиентская часть должна корректно обрабатывать задержки ответа сервера посредством повторных попыток запроса через заданный интервал, отсутствие сетевого подключения с переходом индикатора статуса в соответствующее состояние, а также ошибки серверной части с отображением последних полученных данных и пометкой о времени последнего успешного обновления. Индикатор статуса подключения должен постоянно присутствовать в интерфейсе и поддерживать три состояния: «подключено», «переподключение» и «нет связи».

Сводный перечень нефункциональных требований к клиентской части системы:

­ НФТ-1. Производительность (отклик) – время отклика интерфейса не более 300 мс при 12 виджетах; загрузка списка нарядов не более 500 мс; формирование акта не более 1 с.

­ НФТ-2. Производительность (загрузка) – первоначальная загрузка не более 3 с при скорости подключения от 10 Мбит/с; размер бандла не более 1 МБ в сжатом виде.

­ НФТ-3. Адаптивность (экраны) – корректное отображение от 10" планшетов (минимальная ширина 768 px) до мониторов 1920×1080 и выше.

­ НФТ-4. Адаптивность (сенсор) – сенсорное взаимодействие: drag-and-drop касанием, элементы управления не менее 44×44 px; оптимизация интерфейса нарядов для работы в перчатках.

­ НФТ-5. Надёжность (потеря связи) – сохранение последних данных на экране при потере связи с пометкой о времени обновления; кэширование введённых данных нарядов.

­ НФТ-6. Надёжность (восстановление) – автоматическое возобновление опроса без перезагрузки страницы; автосохранение черновиков при внезапном закрытии приложения.

­ НФТ-7. Удобство использования – визуальное подтверждение действий (toast-уведомления); модальное подтверждение деструктивных операций; единообразная цветовая индикация во всех компонентах.

­ НФТ-8. Расширяемость – добавление новых типов виджетов и регламентных работ без модификации существующих компонентов; инкапсуляция транспортной логики в сервисный слой.

­ НФТ-9. Совместимость – две последние стабильные версии Google Chrome, Mozilla Firefox, Microsoft Edge; поддержка Internet Explorer не требуется.

Нефункциональные требования определяют характеристики производительности, адаптивности, надёжности и расширяемости системы. Время отклика интерфейса на действия пользователя не должно превышать 300 миллисекунд при одновременном отображении до 12 виджетов. Обновление данных виджетов при поступлении новых значений от сервера не должно вызывать блокировку интерфейса и должно выполняться без потери возможности взаимодействия пользователя с другими элементами. Операции перемещения и изменения размера виджетов должны выполняться плавно, с частотой обновления отрисовки не менее 30 кадров в секунду. Время первоначальной загрузки приложения – не более 3 секунд при скорости подключения от 10 Мбит/с и пустом кэше браузера. Размер итогового бандла – не более 1 МБ в сжатом виде. Загрузка списка нарядов (до 100 записей) должна занимать не более 500 миллисекунд. Формирование акта выполненных работ – не более 1 секунды.

Интерфейс должен корректно отображаться на настольных мониторах с разрешением от 1920×1080 и планшетных устройствах с диагональю от 10 дюймов. На планшетных устройствах допускается сокращённое представление информационной панели при сохранении доступа ко всем функциям системы. На устройствах с сенсорным экраном должны быть доступны операции перемещения виджетов касанием, изменения размера и взаимодействия с элементами графиков. Размер интерактивных элементов управления должен составлять не менее 44×44 пикселей. Интерфейс работы с нарядами должен быть оптимизирован для использования на планшетах в условиях производственных площадок, с увеличенными элементами управления для работы в перчатках.

При потере соединения с сервером последние полученные данные должны сохраняться на экране с пометкой о времени последнего обновления, а при восстановлении связи – обновляться автоматически без перезагрузки страницы. В случае разрыва соединения во время работы с нарядом введённые данные должны кэшироваться локально и автоматически синхронизироваться при восстановлении связи. Система должна предотвращать потерю данных при внезапном закрытии приложения путём автоматического сохранения черновиков. Система должна предоставлять визуальное подтверждение для ключевых действий пользователя, а для деструктивных операций – запрашивать подтверждение через модальное окно. Цветовая индикация состояний оборудования и зон метрик должна быть единообразной во всех компонентах интерфейса.

Архитектура приложения должна обеспечивать возможность добавления новых типов виджетов без модификации существующих компонентов: новый тип виджета реализуется как отдельный компонент, регистрируемый в реестре виджетов. Взаимодействие с серверной частью должно быть инкапсулировано в отдельный сервисный слой, позволяющий изменять транспортный протокол без модификации компонентов представления. Управление состоянием приложения должно быть организовано через централизованное хранилище, обеспечивающее единый источник данных для всех компонентов. Поддерживаемые браузеры – две последние стабильные версии Google Chrome, Mozilla Firefox и Microsoft Edge.

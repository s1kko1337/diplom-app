# Информационная система технического обслуживания и ремонта продукции. Мониторинг и визуализация ("ООО Горная машиностроительная компания – Рудгормаш")

Клиентское веб-приложение, разработанное в рамках выпускной квалификационной работы. Представляет собой универсальный инструмент для визуализации и первичного анализа телеметрических данных бурового оборудования в режиме реального времени, а также для цифрового управления процессами технического обслуживания и ремонта (ТОиР).

## Цель работы

Создание универсального инструмента для визуализации и первичного анализа телеметрических данных бурового оборудования в масштабе реального времени, а также для цифрового управления процессами технического обслуживания и ремонта, позволяющего оптимизировать процесс и повысить качество проведения технического обслуживания.

## Задачи, решаемые в работе

1. **Анализ существующих решений** в сфере мониторинга промышленного оборудования и выявление их недостатков.
2. **Проектирование архитектуры** клиентского приложения и схемы взаимодействия с серверной частью.
3. **Разработка функциональных модулей** приложения (мониторинг, оборудование, ТОиР, аналитика, отчётность).
4. **Реализация пользовательского интерфейса** с адаптивной вёрсткой и единым визуальным языком.
5. **Функциональное тестирование системы** (end-to-end сценарии).

## Функциональные возможности

Приложение реализует следующие модули:

| Модуль | Маршрут | Назначение |
| --- | --- | --- |
| **Главная** | `/` | Сводная панель: парк оборудования и ключевые показатели. |
| **Мониторинг** | `/dashboard` | Дашборд телеметрии в реальном времени по всему парку. |
| **Мой дашборд** | `/dashboard/personal` | Настраиваемая сетка виджетов (drag-and-drop, набор показателей под пользователя). |
| **Оборудование** | `/equipment` | Список единиц техники со статусами (работа / простой / авария / отключён). |
| **Карточка оборудования** | `/equipment/:id` | Характеристики, метрики, узлы, история датчиков, графики. |
| **Дашборд единицы** | `/equipment/:id/dashboard` | Детальная телеметрия конкретной единицы оборудования. |
| **Уведомления** | `/alerts` | Журнал событий и срабатываний пороговых значений датчиков. |
| **Аналитика** | `/analytics` | KPI, эффективность использования техники за выбранный период. |
| **Журнал ТС** | `/journal` | Журнал технического состояния оборудования. |
| **Техобслуживание** | `/maintenance` | Канбан-доска нарядов ТОиР (план → в работе → приёмка → завершено). |
| **Создание наряда** | `/maintenance/create` | Формирование наряда по регламенту (ЕО, ТО-1…ТО-3, ТР, КР). |
| **Выполнение и акт** | `/maintenance/:id`, `/maintenance/:id/document` | Пошаговое выполнение работ, фиксация замеров и материалов, генерация акта / чек-листа. |
| **Отчёты** | `/reports` | Создание, редактирование и просмотр отчётов (с разграничением прав). |
| **Настройки** | `/settings` | Параметры интерфейса и пользовательские предпочтения. |

### Ролевая модель

Доступ к функциям разграничен по ролям (RBAC):

- **Инженер** (`engineer`)
- **Механик** (`mechanic`)
- **Мастер** (`foreman`)

Регламент ТОиР соответствует системе планово-предупредительных ремонтов: ежесменное обслуживание (ЕО), ТО-1/ТО-2/ТО-3, текущие ремонты (ТР-1…ТР-3) и капитальный ремонт (КР).

## Архитектура

Это клиентская часть (SPA). Взаимодействие с серверной частью спроектировано через изолированный слой API: компоненты обращаются к функциям модуля `src/api/*`, не зная об источнике данных. В текущей сборке этот слой реализован моками (`src/api/mock/*`) с имитацией сетевых задержек, состояний загрузки и ошибок, поэтому приложение работает автономно, без развёрнутого бэкенда. Для подключения реального сервера достаточно заменить реализации в `src/api/*` на HTTP-запросы, сохранив их сигнатуры.

### Технологический стек

- **Vue 3** (Composition API, `<script setup>`)
- **Vite** — сборка и dev-сервер
- **Pinia** — управление состоянием
- **Vue Router** — маршрутизация и навигационные гварды (авторизация, права)
- **Tailwind CSS 4** + **shadcn-vue** (reka-ui) — UI-компоненты
- **ECharts** (`vue-echarts`) — графики телеметрии
- **grid-layout-plus** — настраиваемая сетка дашбордов
- **Playwright** — end-to-end тестирование

### Структура проекта

```
src/
├── api/            # Слой доступа к данным (контракт + mock-реализация)
│   ├── mock/       #   mock-обработчики и сид-данные
│   └── *.js        #   модули: auth, equipment, sensors, maintenance, reports…
├── components/     # UI-компоненты (по доменам: equipment, maintenance, analytics, ui…)
├── composables/    # Переиспользуемая логика (useLiveData, usePermissions, useTheme…)
├── layouts/        # Макеты (AuthLayout, DefaultLayout)
├── router/         # Конфигурация маршрутов и гварды доступа
├── stores/         # Pinia-хранилища (auth, equipment, sensors, maintenance…)
├── utils/          # Константы, форматтеры, права доступа
└── views/          # Страницы приложения
```

## Запуск

### Требования

- **Node.js** `^20.19.0` или `>=22.12.0`
- npm

### Установка зависимостей

```sh
npm install
```

### Запуск в режиме разработки

```sh
npm run dev
```

Приложение будет доступно по адресу, указанному в выводе Vite (по умолчанию `http://localhost:5173`).

### Вход в систему

На странице входа выберите пользователя из списка и введите **любой пароль** (аутентификация имитируется mock-слоем). Для каждого пользователя задана своя роль, что позволяет проверить разграничение доступа.

### Сборка для production

```sh
npm run build      # сборка в каталог dist/
npm run preview    # локальный предпросмотр production-сборки
```

> Сборка настроена под публикацию на GitHub Pages с базовым путём `/diplom-app/` (включая SPA-фоллбэк через `404.html`). Для размещения по другому пути измените `GH_PAGES_BASE` в `vite.config.js`.

## Тестирование

Функциональное тестирование системы реализовано на Playwright. Сценарии разбиты на функциональные (`ft-*`) и нефункциональные (`nfr-*`) требования, а также на ролевые сценарии.

```sh
npm run test:e2e          # прогон всех e2e-тестов
npm run test:e2e:ui       # интерактивный режим (UI)
npm run test:e2e:headed   # с отображением браузера
npm run test:e2e:report   # открыть HTML-отчёт о последнем прогоне
```

Подробнее о тестовом наборе — в [`tests/e2e/README.md`](tests/e2e/README.md).
